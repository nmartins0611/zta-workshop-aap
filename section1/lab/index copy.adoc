== Overview

Connect Ansible Automation Platform to the Zero Trust stack. Workshop credentials are pre-created with `setup/configure-aap-credentials.yml`; you verify Vault lookups and related objects, then configure inventory, project, templates, and run checks against IdM, Vault, OPA, NetBox, and Gitea.

[IMPORTANT]
.Why this section exists
====
Until AAP is wired to the same **identity, policy, secrets, and inventory truth** the rest of the architecture uses, automation would be a *shadow admin path*—easy to bypass Zero Trust intent. Here you make AAP a **first-class citizen** of the ZTA fabric: it authenticates like other services, reads policy from OPA, pulls secrets from Vault, and builds inventory from NetBox so *what you run jobs against* matches *what the organisation says exists*.
====

== Exercise 1.1 — Verify AAP credentials (pre-configured, Vault-sourced)

=== Goal

Workshop credentials are created with `setup/configure-aap-credentials.yml`. You **confirm** Vault KV data and controller objects match the lab—no manual credential creation in the UI for this exercise.

[TIP]
.Zero Trust: secrets on the platform
====
Machine and Arista passwords are **not** stored in AAP: the controller holds **Vault paths** and resolves them at job time. Only the bootstrap Vault lookup (and NetBox token) behave like classic “stored” secrets in the platform database—tighten that in production (AppRole, TLS, policy).
====

=== Step 1 — Confirm Vault paths

[source,bash]
----
export VAULT_ADDR=http://vault.zta.lab:8200
vault login -method=userpass username=admin password=ansible123!

vault kv get secret/machine/rhel
vault kv get secret/network/arista
vault kv get secret/db/admin
----

=== Step 2 — Confirm credentials in the controller

Open **Credentials** and verify these names exist (from `setup/configure-aap-credentials.yml`). Edit each and confirm:

* `ZTA Vault Credential` — Vault URL and v2; bootstrap auth present.
* `ZTA Machine Credential` — user `rhel`, sudo; **Password** and **Become** use lookups to `secret/data/machine/rhel` (`password`, `become_password`).
* `ZTA Arista Credential` — lookups on `secret/data/network/arista`.
* `ZTA NetBox Credential` — URL and token match NetBox.
* `ZTA Vault SSH Credential` — present (AppRole / signed SSH).

Do not change lookup paths unless an instructor is troubleshooting.

[TIP]
.Gitea credential
====
`configure-aap-credentials.yml` does **not** create **ZTA Gitea Credential**. Confirm or add it when you set up the **ZTA Workshop** project (Exercise 1.3) if Git requires it.
====

Functional checks: **Exercise 1.5** (Verify ZTA Services) and **1.6–1.7** prove lookups from AAP.

=== Step 3 — NetBox type missing?

If the **NetBox** credential type or **ZTA NetBox Credential** is absent, follow the **Manual: NetBox credential type** steps in `section1/README.md`, then re-run automation or recreate the NetBox credential per Exercise 1.3.

=== Step 4 — Vault SSH CA

[source,bash]
----
vault read ssh/config/ca
vault read ssh/roles/ssh-signer
----

.Credential summary
[cols="2,2,3",options="header"]
|===
| Credential | Type | Source
| ZTA Vault Credential | Vault lookup | Bootstrap (stored in AAP)
| ZTA Machine | Machine | KV `secret/data/machine/rhel` at job time
| ZTA Arista | Network | KV `secret/data/network/arista` at job time
| ZTA NetBox | NetBox | Token in AAP
| ZTA Vault SSH | Vault Signed SSH | AppRole + `ssh-signer`
|===

[TIP]
.Vault SSH CA vs static keys
====
**Signed SSH certificates** are the Zero Trust answer to shared static keys. Exercise 1.7 exercises this path end-to-end.
====

== Exercise 1.2 — IdM LDAP (Authentication Methods UI)

=== Goal

On **Ansible Automation Platform 2.6**, LDAP against IdM is often configured under **Access Management** → **Authentication Methods** → **Create authentication** (not only the legacy `PATCH /api/controller/v2/settings/ldap/` path used by `setup/configure-aap-ldap.yml`). This exercise records the **IdM field values** that match the workshop playbooks and `inventory/group_vars/all.yml`.

If IdM login already works, you can **skim** this exercise to validate settings; if logins fail on gateway-based installs, complete it.

=== Prerequisites

* IdM (FreeIPA) on `central.zta.lab` with domain `zta.lab` → base DN `dc=zta,dc=lab`.
* Controller **trusts the IdM CA** for LDAPS (lab: `setup/enroll-idm-clients.yml` on the controller host).
* IdM **`admin`** password: workshop default `ansible123!` (same as `idm_admin_password`).

Adjust hostnames and `dc=` components if your lab uses a different `idm_domain`.

=== Steps — Create LDAP authentication

. Log in as platform **admin**. Open **Access Management** → **Authentication Methods** → **Create authentication**.
. Set **Authentication type** to **LDAP**.
. Fill the top fields:

+
[cols="2,3",options="header"]
|===
| Field | Value (workshop defaults)
| Name | `IdM LDAP` (or any label)
| LDAP Server URI | `ldaps://central.zta.lab`
| LDAP Bind DN | `uid=admin,cn=users,cn=accounts,dc=zta,dc=lab`
| LDAP Bind Password | IdM `admin` password (`ansible123!` unless overridden)
| LDAP Group Type | `GroupOfNamesType` (recommended) or `MemberDNGroupType` — see parameters below
| LDAP User DN Template | Leave **empty**
| LDAP Start TLS | **Off** (LDAPS already encrypts the connection)
|===

. **LDAP Connection Options** (YAML):

+
[source,yaml]
----
OPT_REFERRALS: 0
OPT_NETWORK_TIMEOUT: 30
----

. **LDAP Group Type Parameters** (required). These are the **key/value arguments for the selected group type’s constructor** (`LDAP_GROUP_TYPE_PARAMS`). Invalid keys for that type produce *Invalid option for specified GROUP_TYPE*.

+
* **MemberDNGroupType** — generic type; constructor takes **`member_attr`** and **`name_attr`**. Use this if you want both fields explicitly (matches the doc example `{"name_attr": "cn", "member_attr": "member"}`):

+
[source,yaml]
----
name_attr: cn
member_attr: member
----

. **LDAP User Search** (YAML list — three elements):

+
[source,yaml]
----
- cn=users,cn=accounts,dc=zta,dc=lab
- SCOPE_SUBTREE
- (uid=%(user)s)
----

. **LDAP Group Search** (YAML list — three elements):

+
[source,yaml]
----
- cn=groups,cn=accounts,dc=zta,dc=lab
- SCOPE_SUBTREE
- (objectClass=groupOfNames)
----

. **LDAP User Attribute Map** (required; YAML):

+
[source,yaml]
----
first_name: givenName
last_name: sn
email: mail
----

. Enable **Enabled** (and **Create objects** if you want local user records on first IdM login). Save (**Create Authentication Method**).

=== Organisation and team mapping

`setup/configure-aap-ldap.yml` also sets **organisation map**, **team map**, and **superuser group** (`zta-admins`, `team-infrastructure`, etc.). Those may appear as **separate authenticator maps** (or advanced fields) in Access Management. If IdM users log in but lack roles or organisations, mirror the `ldap_config` block in `setup/configure-aap-ldap.yml` using the https://docs.redhat.com/en/documentation/red_hat_ansible_automation_platform/2.6/html/access_management_and_authentication/[AAP 2.6 Access management and authentication] guide.

==== Which groups you reference (by DN)

These are the IdM groups the workshop uses in **Group** triggers (`dc=zta,dc=lab`). Use the **full DN** in the **Groups** list for each map (one DN per entry; if you combine several groups in one rule, include each DN the UI expects for that operation).

[cols="2,4",options="header"]
|===
| IdM group (`cn`) | Full DN (paste into **Groups**)
| `zta-admins` | `cn=zta-admins,cn=groups,cn=accounts,dc=zta,dc=lab`
| `team-infrastructure` | `cn=team-infrastructure,cn=groups,cn=accounts,dc=zta,dc=lab`
| `team-devops` | `cn=team-devops,cn=groups,cn=accounts,dc=zta,dc=lab`
| `team-security` | `cn=team-security,cn=groups,cn=accounts,dc=zta,dc=lab`
| `team-applications` | `cn=team-applications,cn=groups,cn=accounts,dc=zta,dc=lab`
|===

[NOTE]
.Login vs role grants
====
You are **not** “allowing” only these users to log in **unless** you add a separate **Allow** map that lists those DNs. **Organization**, **Team**, **Superuser**, and similar maps mean: *if* the authenticated user is in that LDAP group, *then* grant the mapped organisation role, team membership, or superuser flag. Everyone else can still log in if LDAP authentication itself permits them; they simply will not receive those grants without a matching map.

For **Allow** maps and other trigger types, see https://docs.redhat.com/en/documentation/red_hat_ansible_automation_platform/2.6/html/access_management_and_authentication/gw-configure-authentication#gw-authenticator-map-types[Authenticator map types] in the AAP 2.6 Access management guide.
====

==== Suggested maps (UI)

Use **Add authentication mapping** on your LDAP authenticator as needed:

* **Organization** + **Group** = `zta-admins` DN → **Default** organization, **Organization Admin**.
* **Superuser** + **Group** = same `zta-admins` DN → platform superuser (**optional**; use only if IdM `zta-admins` should equal full platform admin).
* **Team** + **Group** = each `team-*` DN → the matching AAP team in **Default** with **Team Member** (use the DN from the table above):

+
[cols="2,2",options="header"]
|===
| IdM group (`cn`) | AAP team name (Default org)
| `team-infrastructure` | Infrastructure
| `team-devops` | DevOps
| `team-security` | Security
| `team-applications` | Applications
|===

Paste the **full DN** into the **Groups** field for each **Group** trigger (per map, or combined in one rule if your design merges triggers).

== Exercise 1.3 — Inventory and project

[TIP]
.Source of truth and GitOps
====
**NetBox** is the *infrastructure source of truth* in this workshop: if a host is decommissioned there, it should disappear from automation inventory on sync—reducing “ghost” targets attackers or operators could hit. **Gitea** holds *intent* (playbooks); together they separate **what exists** (CMDB) from **what we want to do** (code), both subject to policy in later sections.
====

=== Project (Gitea)

**Projects** → **Add**:

* Name: `ZTA Workshop`
* Git URL: `http://gitea.zta.lab:3000/zta-workshop/zta-app.git`
* Credential: `ZTA Gitea Credential`
* Save; wait for successful sync.

=== Inventory (NetBox)

. **Inventories** → create `ZTA Lab Inventory`.
. Open the inventory → **Sources** → **Add**:

|===
| Name | NetBox CMDB
| Source | Sourced from a project
| Project | ZTA Workshop
| Inventory file | `inventory/netbox_inventory.yml`
| Credential | ZTA NetBox Credential
|===

. Enable **Overwrite** and **Update on launch**; sync and verify hosts (central, control, vault, netbox, app, db, ceos1–3, etc.).

[TIP]
.Update on launch
====
Refreshing inventory **before each job** reduces stale targeting: policy and playbooks see **current** CMDB state, which is how Zero Trust ties **continuous authorisation** to **current asset posture** (at least at inventory granularity).
====

== Exercise 1.5 — Verify ZTA services

Run **Verify ZTA Services**. Expect healthy IdM, Vault, OPA, NetBox, DNS, PostgreSQL, and Arista facts.

[TIP]
.Never trust, always verify
====
**Service checks** are the operational face of “never trust, always verify”: you do not assume IdM or OPA is up because yesterday it was. For the workshop, this mirrors **continuous diagnostics** on which policy decisions and secret issuance depend.
====

== Exercise 1.6 — Test Vault integration

Run **Test Vault Integration**. Observe short-lived PostgreSQL users (unique name per run, TTL, revoke at end).

[TIP]
.Short-lived database identities
====
Dynamic DB users with a **TTL** limit how long a leaked credential is useful—core **assume breach** hygiene. Vault **revokes** the lease so the database role disappears without a human running `DROP USER`. That is how **least privilege** becomes **time-bound privilege**.
====

== Exercise 1.7 — Test Vault SSH certificates

Run **Test Vault SSH Certificates**. Observe CA-signed cert, TTL, principals, successful SSH to app/db.

Optional manual sign (from README):

[source,bash]
----
export VAULT_ADDR=http://vault.zta.lab:8200
vault login -method=userpass username=admin password=ansible123!
ssh-keygen -t rsa -b 2048 -f /tmp/ephemeral -N '' -q
vault write -field=signed_key ssh/sign/ssh-signer \
  public_key=@/tmp/ephemeral.pub valid_principals=rhel > /tmp/ephemeral-cert.pub
ssh-keygen -L -f /tmp/ephemeral-cert.pub
----

[TIP]
.Cryptographic proof of permission to log in
====
The certificate encodes **who may act as which principal** and **for how long**. Hosts need only trust the CA once; they do not need network reachability to Vault at login time. That separation is useful in segmented networks and matches the ZTA idea that **access is explicitly granted and expires**.
====

== Exercise 1.8 — Test OPA policy

Run **Test OPA Policy**. Expect allow/deny scenarios for patching, VLAN, and DB access per group membership.

[TIP]
.Deny by default (PDP)
====
OPA returns **allow** only when Rego rules fire; everything else is **implicit deny**. **IdM groups** supply *role-like* attributes the PDP can use, but unlike a static RBAC matrix in one app, the same policy can be **versioned, tested, and shared** across AAP and other PEPs. You are seeing **central policy** backing **distributed enforcement**.
====
