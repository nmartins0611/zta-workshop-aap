== Architecture summary

_ZTA concept: *Defence in depth* -- two independent policy rings evaluate both the human (outer) and the workload (inner) before any network change proceeds. An attacker needs to compromise both layers._

* **Outer ring:** user must be in the **Infrastructure** AAP team to launch VLAN/Network templates (`aap.gateway` policy, evaluated by the controller before launch).
* **Inner ring:** trusted SPIFFE ID, IdM `network-admins` group, VLAN 100--999, allowed action (`zta.network` policy, evaluated inside the playbook).

Trusted workload SPIFFE ID (lab): `spiffe://zta.lab/workload/network-automation`

[IMPORTANT]
.Why two rings? Ansible makes defence in depth practical
====
**Humans** can be phished; **scripts** can be stolen. The **outer ring** (AAP Policy as Code) ensures only the right *people* can press "go" on high-risk network automation. The **inner ring** (in-playbook OPA call) adds **workload identity**: the job proves it runs on **the attested automation platform** (SPIRE SVID), not someone's laptop running a copy of the playbook. Ansible makes this practical -- the outer ring is **built into AAP** (zero playbook changes needed), and the inner ring is a **simple `uri` task** in the playbook. Together they implement defence in depth -- an attacker needs both a valid user session *and* a trusted execution environment to change the fabric.
====

== Exercise 4.1 -- Job template

_ZTA concept: *Auditable network change* -- every VLAN creation is tied to both a human identity and a workload identity, recorded in the CMDB for forensics and compliance._

[cols="2,3",options="header"]
|===
| Field | Value
| Name | Configure VLAN
| Playbook | `section4/playbooks/configure-vlan.yml`
| Inventory | ZTA Lab Inventory
| Credentials | ZTA Machine Credential, ZTA Arista Credential
| Survey | `new_vlan_id` (e.g. `200`), `new_vlan_name` (e.g. `DMZ`)
|===

[TIP]
.Ansible + NetBox: automated CMDB audit trail
====
The playbook does not just create a VLAN on switches -- it updates **NetBox** with who initiated the change and which workload identity was bound to the automation. Ansible automates both the **network change** and the **CMDB update** in a single job, so they cannot fall out of sync. This is invaluable for **forensics** ("was this VLAN created by a rogue job?") and for keeping **inventory trustworthy** for the next policy decision.
====

== Exercise 4.2 -- Denied (`neteng`)

_ZTA concept: *Layered deny-by-default* -- an unauthorised user is blocked at the first available gate. If the outer ring is bypassed, the inner ring still catches them._

. Log into AAP as `neteng`.
. Launch the **Configure VLAN** template with VLAN 200 / DMZ.
. Observe the result -- **two denial paths** are possible depending on your LDAP authenticator maps:

**Path A -- Outer ring blocks (expected with correct LDAP maps):** `neteng` has no AAP team membership, so the `aap.gateway` policy denies the launch at the platform level. The playbook never starts and no SPIFFE check occurs. This is the same pattern as Section 3.

**Path B -- Inner ring blocks (if LDAP maps place all users in Infrastructure):** The outer ring passes, the playbook starts, SPIFFE verification **succeeds** (the platform is legitimate), but OPA's inner ring denies because `neteng` is not in the IdM `network-admins` group. This is a deliberate teaching point: **trust the platform** and **authorise the person** are separate questions.

Either path demonstrates deny-by-default. Discuss which ring caught the request and why.

[TIP]
.Ansible + two OPA rings: defence in depth in practice
====
If your outer ring is correctly scoped to **Infrastructure** team only, `neteng` never reaches the inner ring -- AAP blocks the launch entirely. If your maps are broader, the inner ring still catches the missing group membership. That is **defence in depth** -- overlapping checks mean a misconfiguration in one layer does not automatically grant access. Ansible makes this layered approach **operationally simple**: the outer ring requires no playbook changes (it is built into AAP), and the inner ring is just a few tasks in the playbook.
====

== Exercise 4.3 -- Allowed (`netadmin`)

_ZTA concept: *Verified identity + verified workload = authorised change* -- both the human and the automation platform must prove their identity before the network is touched._

As `netadmin`, create VLAN 200 (DMZ). Verify on Arista (`show vlan brief`) and in NetBox (audit fields).

[TIP]
.Ansible: end-to-end network change in one job
====
A single AAP job verifies the workload identity (SPIFFE), checks policy (OPA), configures the switches (Arista `eos_vlans`), and updates the CMDB (NetBox). VLANs are a building block of **micro-segmentation**. Creating them only through Ansible's gated automation reduces **untracked lateral movement** helpers (shadow VLANs) and ties every network change to **identity + workload proof** -- all in one auditable, repeatable job.
====

== Exercise 4.4 -- Edge cases

_ZTA concept: *Runtime parameter validation* -- the inner ring validates data the outer ring could not know at launch time (specific VLAN ID, SPIFFE health), catching invalid or dangerous requests before they reach the network._

The outer ring (AAP gateway policy) answered **"who is allowed to press the button?"** before the job started. But it had no way to know _what_ the user would type into the survey, or whether the automation platform's cryptographic identity would still be valid at execution time. That is the inner ring's job -- it evaluates **runtime data** that only exists once the playbook is running.

This exercise proves the inner ring catches three classes of failure that the outer ring is structurally blind to.

=== 4.4a -- Invalid VLAN ID

. Log in as `netadmin` (who passed the outer ring in Exercise 4.3).
. Launch **Configure VLAN** with VLAN ID `5000` and any name.
. The job starts (the outer ring has no opinion on VLAN IDs), but the OPA network policy **denies** the request because `5000` is outside the permitted range (100–999).

*Why this matters:* An attacker or a careless operator who passes the platform gate could still request a VLAN that conflicts with infrastructure reservations, management networks, or trunk ranges. The outer ring cannot enforce this -- it sees a template name and a user, not the VLAN number. OPA's inner ring evaluates the **actual parameter** at runtime and blocks before any switch is touched.

=== 4.4b -- Missing workload identity (SPIFFE)

. From `central`, stop the SPIRE agent on the automation controller:
+
[source,bash]
----
ssh rhel@control.zta.lab sudo systemctl stop spire-agent
----
+
. Launch **Configure VLAN** again as `netadmin` with a valid VLAN (e.g. `201`).
. The outer ring passes (correct user, correct team), but the first playbook task **fails** -- it cannot fetch an X.509 SVID because the agent is down. No OPA check occurs; no switch is touched.
. Restart the agent after the test:
+
[source,bash]
----
ssh rhel@control.zta.lab sudo systemctl start spire-agent
----

*Why this matters:* Imagine a compromised host running a copy of the playbook. Even if the attacker has valid AAP credentials (stolen token, session hijack), the playbook requires a **cryptographic proof of workload identity** from the local SPIRE agent. A host that is not attested by the SPIRE server -- or one where the agent has been killed by an attacker trying to avoid detection -- simply cannot produce an SVID. The network change is blocked at the source, not at the switch. This is the difference between **"the right person asked"** and **"the right person asked from the right place"**.

=== 4.4c -- Granting group membership (optional)

. Add `neteng` to the `network-admins` IdM group:
+
[source,bash]
----
ipa group-add-member network-admins --users=neteng
----
+
. Log into AAP as `neteng` and retry **Configure VLAN** with VLAN `202`.
. If `neteng` is now in the **Infrastructure** AAP team (or the outer ring allows the Readonly team), the job starts. The inner ring checks IdM groups and finds `network-admins` -- the request is **allowed**.
. Clean up afterward:
+
[source,bash]
----
ipa group-remove-member network-admins --users=neteng
----

*Why this matters:* This demonstrates that policy decisions are **dynamic**, not baked into code. The same playbook, the same template, the same OPA policy -- but a single group membership change in IdM flips the outcome from deny to allow. In a ZTA model, access is **continuously evaluated** against the current state of identity, not a static ACL written months ago.

[TIP]
.Ansible + OPA: why two rings are better than one
====
Each edge case exposes a gap that a single policy layer cannot cover alone:

[cols="1,2,2",options="header"]
!===
! Scenario ! Outer ring (AAP gateway) ! Inner ring (in-playbook OPA)
! Invalid VLAN 5000 ! *Blind* -- sees template name and user, not survey values ! *Catches it* -- evaluates the VLAN ID against allowed ranges
! SPIRE agent down ! *Blind* -- does not know whether the execution host has a valid SVID ! *Catches it* -- playbook fails before reaching OPA because no SVID can be fetched
! Wrong user group ! *May catch it* (if team mapping is strict) ! *Always catches it* -- checks IdM groups regardless of AAP team membership
!===

Ansible makes this layered approach **operationally simple**: the outer ring is built into AAP (zero playbook changes), and the inner ring is a few `uri` and `command` tasks in the playbook. Together they implement **defence in depth** -- a misconfiguration or bypass in one layer does not automatically grant access to the network.
====

== Hands-on: Write the NetBox CMDB update

_ZTA concept: *CMDB as source of truth* -- any infrastructure change not recorded in the CMDB is invisible to compliance, monitoring, and policy engines. An unregistered VLAN is indistinguishable from a rogue one._

The full `configure-vlan.yml` playbook updates NetBox automatically. This exercise isolates the CMDB step so you write it yourself using the NetBox REST API.

[IMPORTANT]
.What to do
====
. Open `section4/playbooks/update-netbox-vlan-student.yml` in a text editor (or view it in Gitea).
. The playbook has three commented-out tasks marked `YOUR CODE STARTS HERE`. Each has hints and `FILL_IN` placeholders.
. Uncomment each task and replace the placeholders:

[cols="1,3",options="header"]
!===
! Task ! What you write
! 1. Check if VLAN exists ! `uri` GET to `{{ netbox_url }}/api/ipam/vlans/?vid={{ new_vlan_id }}` with auth header. Register result to check `.json.count`.
! 2. Create VLAN ! `uri` POST to `{{ netbox_url }}/api/ipam/vlans/` with `vid` (int!), `name`, `status`, `description`. Only run when count is 0.
! 3. Display summary ! `debug` showing the VLAN ID, name, whether it was created or already existed, and the user.
!===

. In AAP, create a job template called **Update NetBox VLAN (Student)** pointing to your skeleton playbook. Use the same inventory, credentials, and EE.
. Run **Configure VLAN** as `netadmin` to create the VLAN on the switches first, then run your template to update NetBox.
. Verify in the NetBox UI (`http://netbox.zta.lab:8880`, navigate to IPAM -> VLANs) that the VLAN appears.
. Run your template again -- it should report "Already exists" instead of creating a duplicate.
====

[TIP]
.Why automate the CMDB update?
====
If the switch change and the CMDB update are separate manual steps, they **will** drift. Someone will create a VLAN and forget to update NetBox, or update NetBox but fat-finger the VLAN ID. Ansible keeps both in the same job, so either both succeed or neither does. In a ZTA architecture, the CMDB feeds downstream policy decisions -- an unregistered VLAN might trigger alerts, block traffic, or cause compliance violations. **Automation is the glue between infrastructure state and the trust decisions that depend on it.**
====

== Discussion prompts

Why verify workload and user? What breaks if SPIRE is down? How does NetBox auditing help?

[TIP]
.Facilitator notes
====
* **User without workload trust** -> stolen credentials are not enough if SPIRE is healthy and policy binds both. Ansible ensures the SPIFFE check is a **mandatory playbook step**, not an optional extra.
* **SPIRE down** -> automation cannot prove workload identity; **fail closed** is preferable to accepting unsigned requests. Ansible's `assert` module enforces this -- the job stops, no partial changes.
* **NetBox** -> Ansible updates the CMDB as part of the same job that changes the network, tying human + workload metadata to **configuration items** for compliance and incident response.
====
