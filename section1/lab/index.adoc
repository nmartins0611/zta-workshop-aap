== Overview

Configure Ansible Automation Platform and connect it to the Zero Trust stack: IdM (identity), Vault (secrets + SSH CA), OPA (policy), NetBox (CMDB), and Gitea (Git). Verify integrations with job templates.

[IMPORTANT]
.Why this section exists
====
Until AAP is wired to the same **identity, policy, secrets, and inventory truth** the rest of the architecture uses, automation would be a *shadow admin path*—easy to bypass Zero Trust intent. Here you make AAP a **first-class citizen** of the ZTA fabric: it authenticates like other services, reads policy from OPA, pulls secrets from Vault, and builds inventory from NetBox so *what you run jobs against* matches *what the organisation says exists*.
====

== Exercise 1.1 — Configure AAP credentials (Vault-sourced)

=== Goal

Bootstrap AAP credentials so machine and device secrets are resolved from Vault at job runtime where applicable.

[TIP]
.Zero Trust: secrets on the platform
====
**Standing credentials in the automation database** are a common weak point: anyone with AAP admin access inherits the keys to the kingdom. Sourcing machine and network passwords from **Vault KV at job time** shrinks that blast radius: AAP stores references, not the secret values. The lone “bootstrap” Vault credential is still a sensitivity—you would harden this in production (AppRole, TLS, namespaced policies)—but the *pattern* is classic ZTA: **no unnecessary persistence of secrets**, and **rotation in one place** (Vault) without re-keying every template.
====

=== Step 1 — Verify Vault paths

[source,bash]
----
export VAULT_ADDR=http://vault.zta.lab:8200
vault login -method=userpass username=admin password=ansible123!

vault kv get secret/machine/rhel
vault kv get secret/network/arista
vault kv get secret/db/admin
----

=== Step 2 — Create `ZTA Vault Credential` (bootstrap)

In the **automation controller**, open **Credentials** (under *Access* or *Automation Execution*, depending on your role) → **Add**:

[cols="2,3",options="header"]
|===
| Field | Value
| Name | `ZTA Vault Credential`
| Type | HashiCorp Vault Secret Lookup
| Vault URL | `http://vault.zta.lab:8200` (Vault IP if pods cannot resolve the name)
| Username | `admin`
| Password | `ansible123!`
| Path to Auth | `userpass`
| API Version | v2
|===

[TIP]
.Why a Vault “lookup” credential
====
This credential exists so AAP can **authenticate to Vault** and resolve other credentials’ fields. It is the narrowest bootstrap the UI exercise needs; production designs still treat it as **highly privileged** and pair it with tight Vault policies, short TTLs, and break-glass procedures (see Section 6).
====

=== Step 3 — Create `ZTA Machine Credential` (KV lookup)

Create a **Machine** credential:

* Username: `rhel`; privilege escalation: `sudo`.
* **Password**: external lookup → `ZTA Vault Credential`, path `secret/machine/rhel` (or backend `secret`, path `machine/rhel`), key `password`.
* **Become password**: same path, key `become_password`.

Do not store the SSH password in AAP.

[IMPORTANT]
====
AAP 2.6 KV v2 lookups: use `secret/machine/rhel`, not `secret/data/machine/rhel` — the controller adds the `data` segment. Using `secret/data/...` can produce `…/secret/data/data/…` (404). The secret is still `vault kv get secret/machine/rhel` on the Vault CLI.
====

[TIP]
.Just-in-time access
====
Each job run **re-fetches** credentials from Vault. That aligns with **just-in-time / just-enough** access: if a password rotates or a policy revokes a path, the next run reflects reality instead of silently using a stale copy stored in AAP’s database.
====

=== Step 4 — Create `ZTA Arista Credential` (KV lookup)

**Network** credential. Use lookups from `secret/network/arista` for `username` and `password` keys (same KV v2 path rules as above).

=== Step 5 — Create `ZTA NetBox Credential`

Type **NetBox** (created by setup if present). URL `http://netbox.zta.lab:8880` and API token as deployed.

If the credential type is missing, create it under **Credential Types** (platform administration) per `section1/README.md` (Input/Injector YAML).

=== Step 6 — `ZTA Vault SSH Credential`

Usually **pre-created** by `setup/configure-aap-credentials.yml` (AppRole → SSH CA signing). Confirm in Vault:

[source,bash]
----
vault read ssh/config/ca
vault read ssh/roles/ssh-signer
----

.Credential summary
[cols="2,2,3",options="header"]
|===
| Credential | Type | Source
| ZTA Vault Credential | Vault lookup | Stored bootstrap password
| ZTA Machine | Machine | KV lookup `secret/machine/rhel`
| ZTA Arista | Network | KV lookup `secret/network/arista`
| ZTA NetBox | NetBox | Token in credential
| ZTA Vault SSH | Vault Signed SSH | AppRole + CA role `ssh-signer`
|===

[TIP]
.Vault SSH CA vs static keys
====
**Signed SSH certificates** are the Zero Trust answer to “everyone shares `~/.ssh/id_rsa`.” Hosts trust the **CA**, not individual keys; AAP proves workload identity to Vault (here via AppRole) and receives a **time-bound** cert. Compromise of an old key matters less when certificates expire and principals are scoped—**no standing SSH identity** on the jump host or in Git.
====

== Reference — Vault paths, lookups, and the credential chain

=== How AAP resolves a Machine password at job launch

. The job is assigned **ZTA Machine Credential** (Machine type).
. For **Password** / **Privilege escalation password**, AAP does **not** read a stored secret; it follows **credential input sources** to **ZTA Vault Credential**.
. **ZTA Vault Credential** logs into Vault (workshop: **userpass** `admin`; **Path to Auth** / default auth path **`userpass`**) and receives a token.
. With that token, the controller reads **KV v2** at the path you configured in the lookup (see table below) and injects the **key** (`password`, `become_password`, etc.) into SSH.

[TIP]
.Controller DNS vs your laptop
====
Credential lookup runs on the **automation controller** (often inside **Kubernetes** pods). If `vault.zta.lab` fails with *Name or service not known*, set the Vault **Server URL** in **ZTA Vault Credential** to the Vault VM **IP** (for example `http://192.168.1.12:8200`). You can still use `vault kv get` from the Vault VM or any host that resolves the name.
====

=== Vault path cheat sheet (CLI vs HTTP vs AAP lookup field)

The same secret is referred to three ways. **AAP 2.6** KV v2 lookups must **not** include a literal `data` segment in the lookup path—the controller adds it when calling the API.

[cols="2,2,2,2",options="header"]
|===
| Purpose | Vault CLI (`vault kv get …`) | HTTP API (writes / direct `curl`) | AAP external lookup **Secret path**
| Linux SSH / sudo (`rhel`) | `secret/machine/rhel` | `POST …/v1/secret/data/machine/rhel` | `secret/machine/rhel` (or backend `secret`, path `machine/rhel`)
| Arista cEOS | `secret/network/arista` | `POST …/v1/secret/data/network/arista` | `secret/network/arista`
|===

[cols="2,3",options="header"]
|===
| Lookup key | Typical use in this lab
| `password` | SSH password for user `rhel` on Linux hosts
| `become_password` | Same value for `sudo` (stored alongside `password` in the same KV path)
| `username` / `password` | Switch login on **ZTA Arista Credential** (from `secret/network/arista`)
|===

[IMPORTANT]
====
Do **not** put `secret/data/machine/rhel` in the AAP lookup path for KV v2: that can produce `…/v1/secret/data/data/machine/rhel` (**404**). Do **not** use only `machine/rhel` without the `secret/` prefix unless your UI has a separate **backend** field set to `secret`—otherwise the controller may treat `machine` as the mount (**404** on `/v1/machine/...`).
====

=== Which credentials attach to jobs (quick reference)

[cols="2,3",options="header"]
|===
| Credential | When you need it on a job template
| **ZTA Machine Credential** | Any template that uses SSH to **Linux** hosts (`rhel`, `become`).
| **ZTA Arista Credential** | Templates (or plays) that run **Arista EOS** modules against **ceos** hosts (`ansible_connection: network_cli`).
| **ZTA NetBox Credential** | **Inventory sync** from NetBox and some playbooks that call NetBox modules—not required on the four Section 1 verification templates unless a playbook uses NetBox APIs with that credential.
| **ZTA Gitea Credential** | **Project** sync from Gitea only—not attached to job templates.
| **ZTA Vault Credential** | Never attached alone to a template; it is consumed via input sources on Machine/Network credentials.
| **ZTA Vault SSH Credential** | Used when a **Machine** credential is configured for **signed SSH** (later hardening flows). The Section 1 `test-vault-ssh.yml` playbook signs certs via **userpass** inside the playbook on the Vault host; it still needs **ZTA Machine Credential** to reach the `vault` host.
|===

Workshop defaults for names (match `setup/configure-aap-project.yml` / `configure-aap-inventory.yml` when using setup playbooks): **Organization** `Default`, **Project** `ZTA Workshop`, **Inventory** `ZTA Lab Inventory`, **Execution environment** `ZTA Workshop EE`.

== Exercise 1.2 — Inventory and project

[TIP]
.Source of truth and GitOps
====
**NetBox** is the *infrastructure source of truth* in this workshop: if a host is decommissioned there, it should disappear from automation inventory on sync—reducing “ghost” targets attackers or operators could hit. **Gitea** holds *intent* (playbooks); together they separate **what exists** (CMDB) from **what we want to do** (code), both subject to policy in later sections.
====

=== Source control credential (Gitea)

Create **ZTA Gitea Credential** before the project (the project cannot sync without it):

**Credentials** → **Add**:

[cols="2,3",options="header"]
|===
| Field | Value
| Name | `ZTA Gitea Credential`
| Type | Source Control
| Username / Password | Workshop Gitea user (defaults are often `gitea` / `ansible123!` if aligned with instructor setup)
|===

Use the URL and account your instructor provides if they differ.

=== Project (Gitea)

**Projects** → **Add**:

* Name: `ZTA Workshop`
* **Execution environment:** `ZTA Workshop EE` (or the custom EE your instructor deployed for this workshop)
* Git URL: `http://gitea.zta.lab:3000/zta-workshop/zta-app.git`
* Credential: `ZTA Gitea Credential`
* Save; wait for successful sync.

=== Inventory (NetBox)

. **Inventories** → create `ZTA Lab Inventory`.
. Open the inventory → **Sources** → **Add**:

|===
| Name | NetBox CMDB
| Source | Sourced from a project
| Project | ZTA Workshop
| Inventory file | `inventory/netbox_inventory.yml`
| Credential | ZTA NetBox Credential
|===

. Enable **Overwrite** and **Update on launch**; sync and verify hosts (central, control, vault, netbox, app, db, ceos1–3, etc.).

[TIP]
.Update on launch
====
Refreshing inventory **before each job** reduces stale targeting: policy and playbooks see **current** CMDB state, which is how Zero Trust ties **continuous authorisation** to **current asset posture** (at least at inventory granularity).
====

== Exercise 1.3 — Job templates

[TIP]
.Why these four templates
====
You are building an **evidence trail** that each pillar works before you trust it for real change. In ZTA terms this is **verify, then enforce**: IdM/Vault/OPA/network paths must be healthy or later sections will fail in ambiguous ways. Treat these jobs as **control-plane health checks** for the PEP (AAP).
====

=== Defaults for every template

Create four **Job templates** (**Templates** → **Create job template**). For **each** template set:

* **Job type:** Run
* **Inventory:** `ZTA Lab Inventory`
* **Project:** `ZTA Workshop`
* **Execution environment:** `ZTA Workshop EE` (workshop image built in `setup/configure-aap-project.yml`; includes collections needed for EOS and general Ansible work)
* **Forks / Timeout:** leave defaults unless your instructor specifies otherwise
* **Privilege escalation:** **On** (templates run `become` on Linux hosts except where the playbook sets `become: false` for a play—AAP should still allow escalation)

=== Credentials per template

Attach credentials on the job template under **Credentials** (you may need **Select** and multiple rows):

[cols="2,3",options="header"]
|===
| Job template | Credentials to attach
| **Verify ZTA Services** | **ZTA Machine Credential** (SSH to `central`, `db`, etc.) **and** **ZTA Arista Credential** (third play gathers **EOS facts** from `ceos1`–`ceos3` via `network_cli`).
| **Test Vault Integration** | **ZTA Machine Credential** only (play targets `vault_servers`; Vault API login uses variables inside the playbook).
| **Test Vault SSH Certificates** | **ZTA Machine Credential** only (Ansible connects to `vault`; signing uses userpass + `vault` CLI on that host; cert-based `ssh` to app/db is run from the Vault host in the playbook).
| **Test OPA Policy** | **ZTA Machine Credential** only (runs on `zta_services` / `central`; HTTP checks to OPA).
|===

Do **not** attach **ZTA Vault Credential** directly to templates—it is only used through **input sources** on Machine/Network credentials.

=== Playbook paths and optional limits

[cols="2,2,1,2",options="header"]
|===
| Template name | Playbook | Suggested limit | Notes
| Verify ZTA Services | `section1/playbooks/verify-zta-services.yml` | _(none)_ | Hits `zta_services`, `db_servers`, and `network` groups.
| Test Vault Integration | `section1/playbooks/test-vault-integration.yml` | `vault` | Speeds failure isolation; optional.
| Test Vault SSH Certificates | `section1/playbooks/test-vault-ssh.yml` | `vault` | Optional; playbook still needs `app` and `db` in inventory for embedded SSH tests.

=== Extra variables (if jobs fail with “undefined variable”)

Static or NetBox-only inventories may not load `inventory/group_vars/all.yml` from the Git repo. If a template errors on missing `vault_addr`, `opa_url`, `idm_admin_password`, etc., add **Extra Variables** on that job template (YAML):

[source,yaml]
----
vault_addr: "http://vault.zta.lab:8200"
vault_skip_verify: true
vault_kv_secrets_path: secret
vault_ssh_secrets_path: ssh
vault_ssh_role_name: ssh-signer
vault_ssh_ttl: 30m
vault_db_secrets_path: database
vault_db_role_name: ztaapp-short-lived
opa_url: "http://central.zta.lab:8181"
netbox_url: "http://netbox.zta.lab:8880"
netbox_token: "0123456789abcdef0123456789abcdef01234567"
idm_admin_password: ansible123!
idm_hostname: central.zta.lab
app_host: app.zta.lab
db_host: db.zta.lab
db_name: ztaapp
----

Adjust hostnames, token, and passwords to match your environment. Use the Vault **IP** in `vault_addr` if the controller cannot resolve `vault.zta.lab`.

== Exercise 1.4 — Verify ZTA services

Run **Verify ZTA Services** using the settings from Exercise 1.3 (**Machine + Arista** credentials, **ZTA Workshop EE**). Expect healthy IdM, Vault, OPA, NetBox, DNS, PostgreSQL, and Arista facts.

[TIP]
.Never trust, always verify
====
**Service checks** are the operational face of “never trust, always verify”: you do not assume IdM or OPA is up because yesterday it was. For the workshop, this mirrors **continuous diagnostics** on which policy decisions and secret issuance depend.
====

== Exercise 1.5 — Test Vault integration

Run **Test Vault Integration** (**ZTA Machine Credential** only, same EE/inventory/project). Observe short-lived PostgreSQL users (unique name per run, TTL, revoke at end). The playbook uses **userpass** (`admin` / `idm_admin_password`) via the Vault API for token, then `vault` CLI on the Vault VM—distinct from the **ZTA Vault Credential** lookup chain used for SSH passwords.

[TIP]
.Short-lived database identities
====
Dynamic DB users with a **TTL** limit how long a leaked credential is useful—core **assume breach** hygiene. Vault **revokes** the lease so the database role disappears without a human running `DROP USER`. That is how **least privilege** becomes **time-bound privilege**.
====

== Exercise 1.6 — Test Vault SSH certificates

Run **Test Vault SSH Certificates** (**ZTA Machine Credential** only). Observe CA-signed cert, TTL, principals, successful SSH to app/db from the Vault host using the ephemeral certificate (the template does **not** require **ZTA Vault SSH Credential** on the job; that credential type is for jobs where the **Machine** credential is wired to signed SSH in the controller).

Optional manual sign (from README):

[source,bash]
----
export VAULT_ADDR=http://vault.zta.lab:8200
vault login -method=userpass username=admin password=ansible123!
ssh-keygen -t rsa -b 2048 -f /tmp/ephemeral -N '' -q
vault write -field=signed_key ssh/sign/ssh-signer \
  public_key=@/tmp/ephemeral.pub valid_principals=rhel > /tmp/ephemeral-cert.pub
ssh-keygen -L -f /tmp/ephemeral-cert.pub
----

[TIP]
.Cryptographic proof of permission to log in
====
The certificate encodes **who may act as which principal** and **for how long**. Hosts need only trust the CA once; they do not need network reachability to Vault at login time. That separation is useful in segmented networks and matches the ZTA idea that **access is explicitly granted and expires**.
====

== Exercise 1.7 — Test OPA policy

Run **Test OPA Policy** (**ZTA Machine Credential** only). Expect allow/deny scenarios for patching, VLAN, and DB access per group membership.

[TIP]
.Deny by default (PDP)
====
OPA returns **allow** only when Rego rules fire; everything else is **implicit deny**. **IdM groups** supply *role-like* attributes the PDP can use, but unlike a static RBAC matrix in one app, the same policy can be **versioned, tested, and shared** across AAP and other PEPs. You are seeing **central policy** backing **distributed enforcement**.
====
