== Overview

Connect Ansible Automation Platform to the Zero Trust stack. Workshop credentials are pre-created with `setup/configure-aap-credentials.yml`; you verify Vault lookups and related objects, then configure inventory, project, templates, and run checks against IdM, Vault, OPA, NetBox, and Gitea.

[IMPORTANT]
.Why this section exists
====
Until AAP is wired to the same **identity, policy, secrets, and inventory truth** the rest of the architecture uses, automation would be a *shadow admin path* -- easy to bypass Zero Trust intent. Here you make AAP a **first-class citizen** of the ZTA fabric: it authenticates like other services, reads policy from OPA, pulls secrets from Vault, and builds inventory from NetBox so *what you run jobs against* matches *what the organisation says exists*.
====

== Controller: IdM LDAP and Policy as Code (prerequisite)

Later sections assume the automation controller accepts **IdM users** for login and enforces **Policy as Code** (OPA gateway) before jobs run. In a full lab build this is usually applied **during deployment** (before Section 1), not in the UI during class.

[TIP]
.Ansible Automation Platform as the integration backbone
====
Each prerequisite is deployed by a **single Ansible playbook** -- no manual clicking across multiple UIs:

* `setup/configure-opa-base.yml` -- load OPA policies (including `aap.gateway`).
* `setup/configure-aap-policy.yml` -- enable controller <-> OPA (Policy as Code).
* `setup/configure-aap-ldap.yml` -- controller **LDAP authentication** against IdM (LDAPS).

This is the **automation-first** principle: Ansible codifies integrations between IdM, OPA, and Vault as **repeatable, auditable playbooks** instead of one-off admin actions that drift over time. Section 1 asks you to **confirm** both integrations so exercises 2--4 behave as written.
====

=== LDAP (controller -> IdM)

* **What it does:** Users log into `https://control.zta.lab` with **workshop IdM accounts** (for example `ztauser` / `ansible123!`), not only the platform `admin` account.
* **Prerequisites:** IdM running; controller trusts the IdM CA (for example after `setup/enroll-idm-clients.yml`). Details: `HANDOFF.md` (*LDAP: controller -> IdM*).

=== Verify LDAP

. Open the automation controller.
. Sign in with an **IdM user** (for example `ztauser`, password `ansible123!` per link:../../docs/lab/common-environment.adoc#lab-common-environment[Lab environment reference]).
. If login fails: complete **Exercise 1.2** (Authentication Methods UI), or confirm `setup/configure-aap-ldap.yml` (legacy API) completed, DNS reaches IdM, and troubleshooting in `docs/deployment-guide.md` / `HANDOFF.md`.

=== Policy as Code (controller -> OPA)

* **What it does:** Before starting a job, the controller can query OPA at `v1/data/aap/gateway/decision` (gateway Rego). Section 3 and 4 depend on this **outer ring**.
* **Automation:** `setup/configure-aap-policy.yml` after `setup/configure-opa-base.yml`.

=== Verify Policy as Code

. As platform **admin**, open **Settings** -> **Policy** / **Open Policy Agent** / **Policy as Code** (exact label varies by patch) and confirm OPA URL, policy path, and enable flags match `docs/deployment-guide.md` Layer 7 (*Manual Policy as Code (OPA gateway)*).
. Or use the API:

[source,bash]
----
curl -sk -u "admin:ansible123!" \
  "https://control.zta.lab/api/controller/v2/settings/opa/" | python3 -m json.tool

Expect `OPA_ENABLED` and `OPA_PRE_ACTION_ENABLED` true; `OPA_POLICY_PATH` is `v1/data/aap/gateway/decision`; `OPA_URL` matches your lab OPA (typically `http://central.zta.lab:8181` per `inventory/group_vars/all.yml`).

If Policy as Code was not applied by automation, use the **manual UI or API** steps in `docs/deployment-guide.md` Layer 7 or run `setup/configure-aap-policy.yml`.

== Exercise 1.1 -- Verify AAP credentials (pre-configured, Vault-sourced)

_ZTA concept: *Secrets management* -- credentials are never stored in the automation platform; they are resolved from a secrets manager at job time, limiting exposure if the platform is compromised._

=== Goal

Workshop credentials are created with `setup/configure-aap-credentials.yml`. You **confirm** Vault KV data and controller objects match the lab -- no manual credential creation in the UI for this exercise.

[TIP]
.Ansible + Vault: just-in-time secrets, zero static passwords
====
Machine and Arista passwords are **not** stored in AAP: the controller holds **Vault paths** and resolves them at job time. AAP's **credential plugin architecture** means playbook authors never hard-code or even see passwords. Credentials flow from Vault to AAP to playbook execution automatically. If Vault rotates a password, the next AAP job picks it up with no playbook changes. Only the bootstrap Vault lookup (and NetBox token) behave like classic "stored" secrets -- tighten those in production with AppRole, TLS, and policy.
====

=== Step 1 -- Confirm Vault paths

[source,bash]
----
export VAULT_ADDR=http://vault.zta.lab:8200
vault login -method=userpass username=admin password=ansible123!

vault kv get secret/machine/rhel
vault kv get secret/network/arista
vault kv get secret/db/admin
----

=== Step 2 -- Confirm credentials in the controller

Open **Credentials** and verify these names exist (from `setup/configure-aap-credentials.yml`). Edit each and confirm:

* `ZTA Vault Credential` -- Vault URL and v2; bootstrap auth present.
* `ZTA Machine Credential` -- user `rhel`, sudo; **Password** and **Become** use lookups to `secret/data/machine/rhel` (`password`, `become_password`).
* `ZTA Arista Credential` -- lookups on `secret/data/network/arista`.
* `ZTA NetBox Credential` -- URL and token match NetBox.
* `ZTA Vault SSH Credential` -- present (AppRole / signed SSH).

Do not change lookup paths unless an instructor is troubleshooting.

[TIP]
.Ansible automates credential wiring at scale
====
`configure-aap-credentials.yml` creates all five credentials, wires Vault lookups, and seeds KV data in one run. In production, this same Ansible-based approach means credential rotation, new service onboarding, and Vault path changes are **version-controlled and repeatable** -- not tribal knowledge in a wiki. Confirm or add the **ZTA Gitea Credential** when you set up the project (Exercise 1.3) if Git requires it.
====

Functional checks: **Exercise 1.5** (Verify ZTA Services) and **1.6--1.7** prove lookups from AAP.

=== Step 3 -- NetBox type missing?

If the **NetBox** credential type or **ZTA NetBox Credential** is absent, follow the **Manual: NetBox credential type** steps in `section1/README.md`, then re-run automation or recreate the NetBox credential per Exercise 1.3.

=== Step 4 -- Vault SSH CA

[source,bash]
----
vault read ssh/config/ca
vault read ssh/roles/ssh-signer
----

.Credential summary
[cols="2,2,3",options="header"]
|===
| Credential | Type | Source
| ZTA Vault Credential | Vault lookup | Bootstrap (stored in AAP)
| ZTA Machine | Machine | KV `secret/data/machine/rhel` at job time
| ZTA Arista | Network | KV `secret/data/network/arista` at job time
| ZTA NetBox | NetBox | Token in AAP
| ZTA Vault SSH | Vault Signed SSH | AppRole + `ssh-signer`
|===

[TIP]
.Ansible + Vault SSH: certificate-based access without key sprawl
====
**Signed SSH certificates** are the Zero Trust answer to shared static keys. Where traditional approaches scatter `authorized_keys` files across hosts, AAP + Vault sign **short-lived certificates** on demand. Ansible manages the trust chain (CA distribution, `sshd` config) so hosts accept certificates automatically. Exercise 1.7 exercises this path end-to-end.
====

== Exercise 1.2 -- IdM LDAP (Authentication Methods UI)

_ZTA concept: *Identity-based access* -- the automation platform authenticates users against a central identity provider so that every action traces back to a verified identity, not a shared admin account._

=== Goal

On **Ansible Automation Platform 2.6**, LDAP against IdM is often configured under **Access Management** -> **Authentication Methods** -> **Create authentication** (not only the legacy `PATCH /api/controller/v2/settings/ldap/` path used by `setup/configure-aap-ldap.yml`). This exercise records the **IdM field values** that match the workshop playbooks and `inventory/group_vars/all.yml`.

If IdM login already works, you can **skim** this exercise to validate settings; if logins fail on gateway-based installs, complete it.

=== Prerequisites

* IdM (FreeIPA) on `central.zta.lab` with domain `zta.lab` -> base DN `dc=zta,dc=lab`.
* Controller **trusts the IdM CA** for LDAPS (lab: `setup/enroll-idm-clients.yml` on the controller host).
* IdM **`admin`** password: workshop default `ansible123!` (same as `idm_admin_password`).

Adjust hostnames and `dc=` components if your lab uses a different `idm_domain`.

=== Steps -- Create LDAP authentication

. Log in as platform **admin**. Open **Access Management** -> **Authentication Methods** -> **Create authentication**.
. Set **Authentication type** to **LDAP**.
. Fill the top fields:

+
[cols="2,3",options="header"]
|===
| Field | Value (workshop defaults)
| Name | `IdM LDAP` (or any label)
| LDAP Server URI | `ldaps://central.zta.lab`
| LDAP Bind DN | `uid=admin,cn=users,cn=accounts,dc=zta,dc=lab`
| LDAP Bind Password | IdM `admin` password (`ansible123!` unless overridden)
| LDAP Group Type | `GroupOfNamesType` (recommended) or `MemberDNGroupType` -- see parameters below
| LDAP User DN Template | Leave **empty**
| LDAP Start TLS | **Off** (LDAPS already encrypts the connection)
|===

. **LDAP Connection Options** (YAML):

+
[source,yaml]
----
OPT_REFERRALS: 0
OPT_NETWORK_TIMEOUT: 30
----

. **LDAP Group Type Parameters** (required). These are the **key/value arguments for the selected group type's constructor** (`LDAP_GROUP_TYPE_PARAMS`). Invalid keys for that type produce *Invalid option for specified GROUP_TYPE*.

+
Per https://django-auth-ldap.readthedocs.io/en/latest/reference.html#django_auth_ldap.config.LDAPGroupType and the LDAP group-type table in https://docs.redhat.com/en/documentation/red_hat_ansible_automation_platform/2.5/html/access_management_and_authentication/gw-configure-authentication#controller-set-up-LDAP[Red Hat AAP 2.5 Access management] (same idea in 2.6):

+
* **GroupOfNamesType** -- IdM `groupOfNames`; equivalent to `MemberDNGroupType('member')`. Only **`name_attr`** is a constructor argument (`member` is fixed):

+
[source,yaml]
----
name_attr: cn
----

+
* **MemberDNGroupType** -- generic type; constructor takes **`member_attr`** and **`name_attr`**. Use this if you want both fields explicitly (matches the doc example `{"name_attr": "cn", "member_attr": "member"}`):

+
[source,yaml]
----
name_attr: cn
member_attr: member
----

+
For FreeIPA/IdM **groupOfNames**, the two options above are **functionally equivalent**.

+
[TIP]
.MemberDNGroupType vs GroupOfNamesType
====
**GroupOfNamesType** + `name_attr` only is the shortest correct form. **MemberDNGroupType** + `name_attr` and `member_attr` is equally valid for IdM when `member_attr` is `member`. Do not pass `member_attr` when **LDAP Group Type** is **GroupOfNamesType** -- that type does not accept it.
====

. **LDAP User Search** (YAML list -- three elements):

+
[source,yaml]
----
- cn=users,cn=accounts,dc=zta,dc=lab
- SCOPE_SUBTREE
- (uid=%(user)s)
----

. **LDAP Group Search** (YAML list -- three elements):

+
[source,yaml]
----
- cn=groups,cn=accounts,dc=zta,dc=lab
- SCOPE_SUBTREE
- (objectClass=groupOfNames)
----

. **LDAP User Attribute Map** (required; YAML):

+
[source,yaml]
----
first_name: givenName
last_name: sn
email: mail
----

. Enable **Enabled** (and **Create objects** if you want local user records on first IdM login). Save (**Create Authentication Method**).

=== Organisation and team mapping

`setup/configure-aap-ldap.yml` also sets **organisation map**, **team map**, and **superuser group** (`zta-admins`, `team-infrastructure`, etc.). Those may appear as **separate authenticator maps** (or advanced fields) in Access Management. If IdM users log in but lack roles or organisations, mirror the `ldap_config` block in `setup/configure-aap-ldap.yml` using the https://docs.redhat.com/en/documentation/red_hat_ansible_automation_platform/2.6/html/access_management_and_authentication/[AAP 2.6 Access management and authentication] guide.

==== Which groups you reference (by DN)

These are the IdM groups the workshop uses in **Group** triggers (`dc=zta,dc=lab`). Use the **full DN** in the **Groups** list for each map (one DN per entry; if you combine several groups in one rule, include each DN the UI expects for that operation).

[cols="2,4",options="header"]
|===
| IdM group (`cn`) | Full DN (paste into **Groups**)
| `zta-admins` | `cn=zta-admins,cn=groups,cn=accounts,dc=zta,dc=lab`
| `team-infrastructure` | `cn=team-infrastructure,cn=groups,cn=accounts,dc=zta,dc=lab`
| `team-devops` | `cn=team-devops,cn=groups,cn=accounts,dc=zta,dc=lab`
| `team-security` | `cn=team-security,cn=groups,cn=accounts,dc=zta,dc=lab`
| `team-applications` | `cn=team-applications,cn=groups,cn=accounts,dc=zta,dc=lab`
|===

[NOTE]
.Login vs role grants
====
You are **not** "allowing" only these users to log in **unless** you add a separate **Allow** map that lists those DNs. **Organization**, **Team**, **Superuser**, and similar maps mean: *if* the authenticated user is in that LDAP group, *then* grant the mapped organisation role, team membership, or superuser flag. Everyone else can still log in if LDAP authentication itself permits them; they simply will not receive those grants without a matching map.

For **Allow** maps and other trigger types, see https://docs.redhat.com/en/documentation/red_hat_ansible_automation_platform/2.6/html/access_management_and_authentication/gw-configure-authentication#gw-authenticator-map-types[Authenticator map types] in the AAP 2.6 Access management guide.
====

==== Suggested maps (UI)

Use **Add authentication mapping** on your LDAP authenticator. Create them in this order (order matters -- the organisation membership map must exist so users land in the Default org on first login):

* **Organization** + trigger **Always** -> **Default** organization, **Member**. This ensures every LDAP user gets basic org membership on login. Without it, users authenticate successfully but see an empty platform with no inventories, templates, or credentials visible.
* **Organization** + trigger **Group** = `zta-admins` DN -> **Default** organization, **Organization Admin**. Grants admin-level org access to the ZTA admins group.
* **Superuser** + trigger **Group** = same `zta-admins` DN -> platform superuser (**optional**; use only if IdM `zta-admins` should equal full platform admin).
* **Team** + trigger **Group** = each `team-*` DN -> the matching AAP team in **Default** with **Team Member** (use the DN from the table above):

+
[cols="2,2",options="header"]
|===
| IdM group (`cn`) | AAP team name (Default org)
| `team-infrastructure` | Infrastructure
| `team-devops` | DevOps
| `team-security` | Security
| `team-applications` | Applications
|===

Paste the **full DN** into the **Groups** field for each **Group** trigger (per map, or combined in one rule if your design merges triggers).

== Exercise 1.3 -- Inventory and project

_ZTA concept: *Authoritative asset inventory* -- automation targets come from a verified CMDB (NetBox), not static files that drift. Playbook intent lives in version-controlled Git (Gitea), not ad-hoc scripts on laptops._

[TIP]
.Ansible + NetBox + Gitea: single source of truth for what and how
====
**NetBox** is the *infrastructure source of truth*: if a host is decommissioned there, it disappears from AAP inventory on sync -- reducing "ghost" targets attackers or operators could hit. **Gitea** holds *intent* (playbooks). AAP's **dynamic inventory** plugin pulls from NetBox automatically, so Ansible jobs always target **current, verified infrastructure**. Together they separate **what exists** (CMDB) from **what we want to do** (code), both subject to policy in later sections.
====

=== Project (Gitea)

**Projects** -> **Add**:

* Name: `ZTA Workshop`
* Git URL: `http://gitea.zta.lab:3000/zta-workshop/zta-app.git`
* Credential: `ZTA Gitea Credential`
* Save; wait for successful sync.

=== Inventory (NetBox)

. **Inventories** -> create `ZTA Lab Inventory`.
. Open the inventory -> **Sources** -> **Add**:

|===
| Name | NetBox CMDB
| Source | Sourced from a project
| Project | ZTA Workshop
| Inventory file | `inventory/netbox_inventory.yml`
| Credential | ZTA NetBox Credential
|===

. Enable **Overwrite** and **Update on launch**; sync and verify hosts (central, control, vault, netbox, app, db, ceos1--3, etc.).

[TIP]
.Ansible dynamic inventory: continuous asset verification
====
Refreshing inventory **before each job** reduces stale targeting: policy and playbooks see **current** CMDB state. AAP's **Update on launch** setting makes this automatic -- every job starts with a fresh NetBox sync. This is how Zero Trust ties **continuous authorisation** to **current asset posture** without manual intervention.
====

== Exercise 1.5 -- Verify ZTA services

_ZTA concept: *Continuous diagnostics* -- never assume a service is healthy because it was yesterday. Verify before every trust decision._

Run **Verify ZTA Services**. Expect healthy IdM, Vault, OPA, NetBox, DNS, PostgreSQL, and Arista facts.

[TIP]
.Ansible multi-host verification in one job
====
A single Ansible playbook reaches IdM, Vault, OPA, NetBox, switches, and databases in parallel -- validating the **entire ZTA control plane** in seconds. Without AAP, you would SSH into each service, run different CLI tools, and mentally correlate the results. AAP provides a **single pane** with timestamped output and pass/fail status across all components.
====

== Exercise 1.6 -- Test Vault integration

_ZTA concept: *Ephemeral credentials* -- short-lived database users limit blast radius if credentials are leaked. Vault issues and revokes them automatically._

Run **Test Vault Integration**. Observe short-lived PostgreSQL users (unique name per run, TTL, revoke at end).

[TIP]
.Ansible + Vault: automated credential lifecycle
====
Dynamic DB users with a **TTL** limit how long a leaked credential is useful -- core **assume breach** hygiene. Ansible automates the full lifecycle: **request** credentials from Vault, **use** them in the playbook, and **revoke** the lease when done. No human runs `DROP USER` -- Vault handles expiry, and AAP orchestrates the flow. That is how **least privilege** becomes **time-bound privilege** at scale.
====

== Exercise 1.7 -- Test Vault SSH certificates

_ZTA concept: *Cryptographic identity* -- SSH access is granted via signed, short-lived certificates instead of static keys scattered across hosts._

Run **Test Vault SSH Certificates**. Observe CA-signed cert, TTL, principals, successful SSH to app/db.

Optional manual sign (from README):

[source,bash]
----
export VAULT_ADDR=http://vault.zta.lab:8200
vault login -method=userpass username=admin password=ansible123!
ssh-keygen -t rsa -b 2048 -f /tmp/ephemeral -N '' -q
vault write -field=signed_key ssh/sign/ssh-signer \
  public_key=@/tmp/ephemeral.pub valid_principals=rhel > /tmp/ephemeral-cert.pub
ssh-keygen -L -f /tmp/ephemeral-cert.pub
----

[TIP]
.Ansible + Vault SSH: no more authorized_keys management
====
The certificate encodes **who may act as which principal** and **for how long**. Hosts need only trust the CA once; they do not need network reachability to Vault at login time. Ansible distributes the CA trust and configures `sshd` across all hosts consistently -- a task that would be error-prone and inconsistent if done manually. In production, AAP jobs use Vault-signed certificates transparently, eliminating the `authorized_keys` sprawl that is a common attack surface.
====

== Exercise 1.8 -- Test OPA policy

_ZTA concept: *Deny by default* -- no action is permitted unless policy explicitly allows it. Policy is centralised, version-controlled, and machine-evaluated._

Run **Test OPA Policy**. Expect allow/deny scenarios for patching, VLAN, and DB access per group membership.

[TIP]
.Ansible + OPA: policy decisions at automation speed
====
OPA returns **allow** only when Rego rules fire; everything else is **implicit deny**. Ansible playbooks query OPA via simple HTTP calls, making policy checks a natural step in any workflow. Unlike a static RBAC matrix in one application, the same OPA policy can be **versioned in Git, tested in CI, and shared** across AAP, CI/CD pipelines, and API gateways. You are seeing **central policy** backing **distributed enforcement** -- with AAP as a first-class consumer.
====
