== Overview

Configure Ansible Automation Platform and connect it to the Zero Trust stack: IdM (identity), Vault (secrets + SSH CA), OPA (policy), NetBox (CMDB), and Gitea (Git). Verify integrations with job templates.

[IMPORTANT]
.Why this section exists
====
Until AAP is wired to the same **identity, policy, secrets, and inventory truth** the rest of the architecture uses, automation would be a *shadow admin path*‚Äîeasy to bypass Zero Trust intent. Here you make AAP a **first-class citizen** of the ZTA fabric: it authenticates like other services, reads policy from OPA, pulls secrets from Vault, and builds inventory from NetBox so *what you run jobs against* matches *what the organisation says exists*.
====

== Exercise 1.1 ‚Äî Configure AAP credentials (Vault-sourced)

=== Goal

Bootstrap AAP credentials so machine and device secrets are resolved from Vault at job runtime where applicable.

[TIP]
.Zero Trust: secrets on the platform
====
**Standing credentials in the automation database** are a common weak point: anyone with AAP admin access inherits the keys to the kingdom. Sourcing machine and network passwords from **Vault KV at job time** shrinks that blast radius: AAP stores references, not the secret values. The lone ‚Äúbootstrap‚Äù Vault credential is still a sensitivity‚Äîyou would harden this in production (AppRole, TLS, namespaced policies)‚Äîbut the *pattern* is classic ZTA: **no unnecessary persistence of secrets**, and **rotation in one place** (Vault) without re-keying every template.
====

=== Step 1 ‚Äî Verify Vault paths

[source,bash]
----
export VAULT_ADDR=http://vault.zta.lab:8200
vault login -method=userpass username=admin password=ansible123!

vault kv get secret/machine/rhel
vault kv get secret/network/arista
vault kv get secret/db/admin
----

=== Step 2 ‚Äî Create `ZTA Vault Credential` (bootstrap)

Resources ‚Üí Credentials ‚Üí Add:

[cols="2,3",options="header"]
|===
| Field | Value
| Name | `ZTA Vault Credential`
| Type | HashiCorp Vault Secret Lookup
| Vault URL | `http://vault.zta.lab:8200`
| Username | `admin`
| Password | `ansible123!`
| API Version | v2
|===

[TIP]
.Why a Vault ‚Äúlookup‚Äù credential
====
This credential exists so AAP can **authenticate to Vault** and resolve other credentials‚Äô fields. It is the narrowest bootstrap the UI exercise needs; production designs still treat it as **highly privileged** and pair it with tight Vault policies, short TTLs, and break-glass procedures (see Section 6).
====

=== Step 3 ‚Äî Create `ZTA Machine Credential` (KV lookup)

Create a **Machine** credential:

* Username: `rhel`; privilege escalation: `sudo`.
* **Password**: use external lookup ‚Üí Credential `ZTA Vault Credential`, path `secret/data/machine/rhel`, key `password`.
* **Become password**: same path, key `become_password`.

Do not store the SSH password in AAP.

[TIP]
.Just-in-time access
====
Each job run **re-fetches** credentials from Vault. That aligns with **just-in-time / just-enough** access: if a password rotates or a policy revokes a path, the next run reflects reality instead of silently using a stale copy stored in AAP‚Äôs database.
====

=== Step 4 ‚Äî Create `ZTA Arista Credential` (KV lookup)

**Network** credential. Use lookups from `secret/data/network/arista` for `username` and `password` keys.

=== Step 5 ‚Äî Create `ZTA NetBox Credential`

Type **NetBox** (created by setup if present). URL `http://netbox.zta.lab:8880` and API token as deployed.

If the credential type is missing, create it under Administration ‚Üí Credential Types per `section1/README.md` (Input/Injector YAML).

=== Step 6 ‚Äî `ZTA Vault SSH Credential`

Usually **pre-created** by `setup/configure-aap-credentials.yml` (AppRole ‚Üí SSH CA signing). Confirm in Vault:

[source,bash]
----
vault read ssh/config/ca
vault read ssh/roles/ssh-signer
----

.Credential summary
[cols="2,2,3",options="header"]
|===
| Credential | Type | Source
| ZTA Vault Credential | Vault lookup | Stored bootstrap password
| ZTA Machine | Machine | KV `secret/data/machine/rhel`
| ZTA Arista | Network | KV `secret/data/network/arista`
| ZTA NetBox | NetBox | Token in credential
| ZTA Vault SSH | Vault Signed SSH | AppRole + CA role `ssh-signer`
|===

[TIP]
.Vault SSH CA vs static keys
====
**Signed SSH certificates** are the Zero Trust answer to ‚Äúeveryone shares `~/.ssh/id_rsa`.‚Äù Hosts trust the **CA**, not individual keys; AAP proves workload identity to Vault (here via AppRole) and receives a **time-bound** cert. Compromise of an old key matters less when certificates expire and principals are scoped‚Äî**no standing SSH identity** on the jump host or in Git.
====

== Exercise 1.2 ‚Äî Inventory and project

[TIP]
.Source of truth and GitOps
====
**NetBox** is the *infrastructure source of truth* in this workshop: if a host is decommissioned there, it should disappear from automation inventory on sync‚Äîreducing ‚Äúghost‚Äù targets attackers or operators could hit. **Gitea** holds *intent* (playbooks); together they separate **what exists** (CMDB) from **what we want to do** (code), both subject to policy in later sections.
====

[TIP]
.Update on launch
====
Refreshing inventory **before each job** reduces stale targeting: policy and playbooks see **current** CMDB state, which is how Zero Trust ties **continuous authorisation** to **current asset posture** (at least at inventory granularity).
====

== Exercise 1.4 ‚Äî Verify ZTA services

Run **Verify ZTA Services**. Expect healthy IdM, Vault, OPA, NetBox, DNS, PostgreSQL, and Arista facts.

[TIP]
.Never trust, always verify
====
**Service checks** are the operational face of ‚Äúnever trust, always verify‚Äù: you do not assume IdM or OPA is up because yesterday it was. For the workshop, this mirrors **continuous diagnostics** on which policy decisions and secret issuance depend.
====

== Exercise 1.5 ‚Äî Test Vault integration

Run **Test Vault Integration**. Observe short-lived PostgreSQL users (unique name per run, TTL, revoke at end).

[TIP]
.Short-lived database identities
====
Dynamic DB users with a **TTL** limit how long a leaked credential is useful‚Äîcore **assume breach** hygiene. Vault **revokes** the lease so the database role disappears without a human running `DROP USER`. That is how **least privilege** becomes **time-bound privilege**.
====

== Exercise 1.6 ‚Äî Test Vault SSH certificates

Run **Test Vault SSH Certificates**. Observe CA-signed cert, TTL, principals, successful SSH to app/db.

Optional manual sign (from README):

[source,bash]
----
export VAULT_ADDR=http://vault.zta.lab:8200
vault login -method=userpass username=admin password=ansible123!
ssh-keygen -t rsa -b 2048 -f /tmp/ephemeral -N '' -q
vault write -field=signed_key ssh/sign/ssh-signer \
  public_key=@/tmp/ephemeral.pub valid_principals=rhel > /tmp/ephemeral-cert.pub
ssh-keygen -L -f /tmp/ephemeral-cert.pub
----

[TIP]
.Cryptographic proof of permission to log in
====
The certificate encodes **who may act as which principal** and **for how long**. Hosts need only trust the CA once; they do not need network reachability to Vault at login time. That separation is useful in segmented networks and matches the ZTA idea that **access is explicitly granted and expires**.
====

== Exercise 1.7 ‚Äî Test OPA policy

Run **Test OPA Policy**. Expect allow/deny scenarios for patching, VLAN, and DB access per group membership.

[TIP]
.Deny by default (PDP)
====
OPA returns **allow** only when Rego rules fire; everything else is **implicit deny**. **IdM groups** supply *role-like* attributes the PDP can use, but unlike a static RBAC matrix in one app, the same policy can be **versioned, tested, and shared** across AAP and other PEPs. You are seeing **central policy** backing **distributed enforcement**.
====
